POST/sessions/:id

Create or initialize a WhatsApp session. Returns connection status, QR when available, and the linked WhatsApp number once connected.

Path Params

Field Type Required Description
id string yes Session identifier, example: 101:sales-1

Response 200

{
  "sessionId": "101:sales-1",
  "status": "connected",
  "qr": null,
  "qrImage": null,
  "error": null,
  "message": "Connected",
  "connectedNumber": "601121000099",
  "me": {
    "id": "601121000099@s.whatsapp.net",
    "phone": "601121000099"
  }
}

connectedNumber is the canonical field for the linked WhatsApp number. Use me.phone or me.id only as fallbacks if needed.

cURL Example

curl -X POST "https://YOUR-DOMAIN/sessions/101:sales-1"

POST/messages/send

Send a WhatsApp message from a connected session.

Request Body

Field Type Required Description
sessionId string yes Connected session ID
to string yes Phone number or full JID. If number only, server appends @s.whatsapp.net
text string no* Message text. Optional if sending media. Used as caption for images/videos.
audioUrl string no URL to an audio file. If provided, sends an audio message.
ptt boolean no If true and audioUrl is provided, sends as a Voice Note (Push-to-Talk).
imageUrl string no URL to an image. Sends as an image message.
videoUrl string no URL to a video. Sends as a video message.
documentUrl string no URL to a document (e.g. PDF). Sends as a document message.
fileName string no File name for document messages (default: "document" or text).
mimetype string no Override default mime type (e.g. "application/pdf" for docs, "audio/ogg; codecs=opus" for voice).
replyTo string no Message ID to quote/reply.

* At least one of text, audioUrl, imageUrl, videoUrl, or documentUrl is required.

Examples

Send Text

{
  "sessionId": "101:sales-1",
  "to": "14155552671",
  "text": "Hello from Baileys API"
}

Send Voice Note (PTT)

{
  "sessionId": "101:sales-1",
  "to": "14155552671",
  "audioUrl": "https://example.com/voice.ogg",
  "ptt": true
}

Send Image with Caption

{
  "sessionId": "101:sales-1",
  "to": "14155552671",
  "imageUrl": "https://example.com/photo.jpg",
  "text": "Check this out!"
}

Response 200

{
  "status": "sent",
  "result": {
    "key": {
      "id": "3EB0....",
      "remoteJid": "14155552671@s.whatsapp.net",
      "fromMe": true
    }
  }
}

cURL Example

curl -X POST "https://YOUR-DOMAIN/messages/send" \
  -H "Content-Type: application/json" \
  -d '{
    "sessionId":"101:sales-1",
    "to":"14155552671",
    "text":"Hi there"
  }'

POST/simulate/inbound

Inject a fake inbound WhatsApp text into the same Postgres flow used by real incoming messages. Useful for testing lead creation, message persistence, and thread trigger assignment.

Request Body

Field Type Required Description
senderPhone string yes Simulated customer phone number
recipientPhone string yes* Connected WhatsApp number to receive the simulated message
sessionId string yes* Alternative to recipientPhone; identifies the session directly
text string no Message body. Defaults to a generated simulation text.
pushName string no Simulated customer display name
messageId string no Optional custom external message ID for repeatable testing

* Provide either recipientPhone or sessionId.

Response 200

{

  "sessionId": "1:my-0182920127",
  "recipientPhone": "60182920127",
  "senderPhone": "60126881912",
  "messageId": "sim_...",
  "inbox": {
    "process_status": "processed"
  },
  "message": {
    "lead_id": 31,
    "thread_id": 44
  },
  "triggerWorked": true
}

POST/groups/create

Create a new WhatsApp group using an already connected session. The server calls Baileys `groupCreate(...)` under the hood and returns the raw WhatsApp group object in the response. This endpoint blocks duplicate groups when the exact member list already exists, regardless of group name. An optional group profile image can be applied after creation. The response always includes groupUid, which is the same value as group.id.

What This Endpoint Does

  • Uses the connected WhatsApp account in sessionId to create the group.
  • Normalizes participant inputs, so raw numbers like 601121000099 become 601121000099@s.whatsapp.net.
  • Returns the created group metadata, including the group JID, subject, owner, and participants.
  • Returns groupUid so clients can reference the group without digging into nested metadata.
  • Checks existing participating groups and blocks creation when the member set matches exactly.
  • Optionally applies a group profile image if profileImageUrl or profileImage is provided.

Important Notes

  • The session must already be connected before you call this endpoint.
  • participants must be a non-empty array of strings or numbers.
  • The session owner becomes the group creator and superadmin.
  • Group name is not used for duplicate detection.
  • Profile image is optional and is applied only after the group is created successfully.
  • profileImageUrl may be a remote URL or a data URL. The server also accepts profileImage as an alias. Duplicate groups are never modified.
  • If WhatsApp rejects one of the participant numbers, the request will fail with a server error returned from Baileys/WhatsApp.

Request Body

Field Type Required Description
sessionId string yes The internal Baileys session ID that is already connected
subject string yes The group name shown to WhatsApp members
participants array of strings yes Array of WhatsApp numbers or JIDs to add to the group. Raw numbers are auto-converted to @s.whatsapp.net.
profileImageUrl string no Optional remote image URL or data URL used as the new group profile picture after creation.

Request Rules

Rule Details
Session state The session must exist in memory and be connected (instance.sock.user must be present).
Participant format Accepts either raw mobile numbers like 601121000099 or full JIDs like 601121000099@s.whatsapp.net.
Normalization The server trims whitespace and appends @s.whatsapp.net when the suffix is missing.

Execution Flow

Step Behavior
1. Validate request Checks that sessionId, subject, and participants are present and that participants is an array.
2. Normalize members Trims values, converts raw numbers to @s.whatsapp.net, and canonicalizes the list for comparison.
3. Detect duplicates Fetches existing participating groups from the connected session and compares exact member sets. Group name is ignored.
4. Create group If no duplicate exists, Baileys creates the group with groupCreate(...).
5. Apply image If profileImageUrl or profileImage is provided, the image is downloaded and set on the new group.

Example Request

{
  "sessionId": "eternalgy-auth",
  "subject": "Bailey Group Test",
  "participants": [
    "601121000099",
    "60182920127"
  ],
  "profileImageUrl": "https://example.com/group-image.jpg"
}

Response 200

{
  "status": "created",
  "groupUid": "120363424859950856@g.us",
  "group": {
    "id": "120363424859950856@g.us",
    "addressingMode": "lid",
    "subject": "Bailey Group Test",
    "owner": "165863617507389@lid",
    "ownerPn": "601158942400@s.whatsapp.net",
    "size": 3,
    "participants": [
      {
        "id": "165863617507389@lid",
        "phoneNumber": "601158942400@s.whatsapp.net",
        "admin": "superadmin"
      },
      {
        "id": "148963021828103@lid",
        "phoneNumber": "60182920127@s.whatsapp.net",
        "admin": null
      },
      {
        "id": "215349811249264@lid",

Response 409

{
  "error": "A group with the same member list already exists (groupUid: 120363424859950856@g.us)",
  "status": "duplicate",
  "groupUid": "120363424859950856@g.us",

Common Errors

Status When It Happens
400 Missing sessionId, subject, or participants, or participants is not an array.
400 The session exists but is not connected yet.
409 An existing WhatsApp group already has the same member list, so the server returns the existing group instead of creating a duplicate.
500 Baileys or WhatsApp rejects the request, or the group create operation fails for another runtime reason. Image upload errors are reported in the response without blocking group creation.

cURL Example

curl -X POST "https://YOUR-DOMAIN/groups/create" \
  -H "Content-Type: application/json" \
  -d '{
    "sessionId":"eternalgy-auth",
    "subject":"Bailey Group Test",
    "participants":["601121000099","60182920127"]
  }'

DELETE/groups/:jid

Make the connected session leave a WhatsApp group. This is the closest supported action to deleting a group. WhatsApp does not provide a true server-side "delete for everyone" endpoint, so this route exits the group for the connected account and returns the group UID and metadata.

What This Endpoint Does

  • Loads the target group metadata before leaving.
  • Calls Baileys groupLeave(...) for the connected session.
  • Returns the group UID, subject, participants, and size in the response.
  • Uses the same group JID you received from POST /groups/create.

Important Notes

  • This is a leave/exit action, not a permanent WhatsApp deletion.
  • The :jid path param should be the group UID, usually something like 120363424859950856@g.us.
  • URL-encode the @ when calling the endpoint from a browser or cURL.
  • The session must already be connected.

Request Query

Field Type Required Description
sessionId string yes Connected Baileys session ID

Path Param

Field Type Required Description
jid string yes The group UID, such as 120363424859950856@g.us.

Response 200

{
  "status": "left",
  "operation": "group_leave",
  "groupUid": "120363424859950856@g.us",
  "group": {
    "id": "120363424859950856@g.us",
    "subject": "Bailey Group Test",
    "participants": [...],
    "size": 3
  },
  "message": "Connected session left the group. WhatsApp does not support permanent server-side deletion of a group."
}

Common Errors

Status When It Happens
400 sessionId or jid is missing, or the session is not connected.
404 The session ID does not exist in memory.
500 Baileys rejects the leave request or the group metadata lookup fails.

cURL Example

curl -X DELETE "https://YOUR-DOMAIN/groups/120363424859950856%40g.us?sessionId=eternalgy-auth"
}

Postgres Message Schema (Inbound Write)

On incoming messages, server writes to et_messages when DATABASE_URL is set. Session and tenant are resolved via et_channel_sessions lookup by session_identifier.

Note: chat/message read endpoints are backed by in-memory cache and WhatsApp history sync events. For long-term complete history, query your Postgres tables.

Insert Mapping

DB Column Value Source
tenant_id Looked up from et_channel_sessions by session_identifier
lead_id Matched from et_leads: first by whatsapp_lid, then by phone digits in external_id
channel_session_id et_channel_sessions.id for the matched session
channel "whatsapp"
external_message_id Baileys msg.key.id
direction "inbound"
message_type Derived: text/image/video/audio/document/sticker/reaction/unknown
text_content conversation/ext text/caption where available
media_url null (current)
raw_payload Full Baileys message JSON (jsonb)
delivery_status "received"

Lead Resolution

Leads are matched in order: (1) by whatsapp_lid if the sender is an @lid JID — fast direct lookup for pre-verified leads, (2) by phone digits extracted from external_id — fallback for leads not yet verified via POST /leads/verify-whatsapp.

Idempotency

Uses DB conflict handling on unique key (tenant_id, channel, external_message_id).

Error Model

Most errors use JSON with error string.

{
  "error": "Missing required fields"
}

Common Cases

Status Case Example
400 Invalid request body {"error":"Missing required fields"}
404 QR unavailable {"error":"QR not ready or already connected"}
500 Internal/server error {"error":"Socket not initialized"}
